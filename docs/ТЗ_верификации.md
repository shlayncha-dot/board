# Техническое задание: верификация КД (DXF/PDF)

## 1. Цель
Реализовать проверку, что для строк спецификации КД в индексированной файловой базе существуют необходимые файлы DXF и PDF, а также корректно отображать в UI:
- отсутствующие файлы;
- дубликаты;
- итоговый отчёт в модальном окне.

## 2. Входные данные

### 2.1 Из Excel/таблицы
Для каждой строки передаются поля:
- `Обозначение` (обязательное для DXF/PDF проверки);
- `Наименование` (для отображения в отчёте PDF);
- `Тип` (для фильтрации DXF-проверки).

### 2.2 Настройки типов
Передаётся массив правил:
- `type` = 1..5;
- `condition` — список префиксов (через пробел/запятую/`;`/перенос строки), по которым выбирается алгоритм поиска PDF.

### 2.3 Индекс файлов
Сервер получает список файлов из `FileIndexStore`:
- `FileName`;
- `Extension`;
- `RelativePath`.

## 3. Функциональные требования

### 3.1 DXF проверка
1. Проверяются только строки, где `Тип` ∈ {`деталь`, `деталь_кон`, `деталь_св`}.
2. Для каждой такой строки файл `*.dxf` ищется по точному совпадению имени без расширения с `Обозначением`.
3. Если найдено:
   - 0 файлов → `Missing`;
   - >1 файла → `Duplicate`;
   - 1 файл → OK.

### 3.2 PDF проверка
1. Для всех строк с непустым `Обозначением` определяется `type` по префиксу до первой точки.
2. Для `type` вызывается соответствующий алгоритм (1..5), иначе точный поиск.
3. Правила статусов как у DXF: 0 — `Missing`, >1 — `Duplicate`, 1 — OK.

### 3.3 API контракты
`POST /api/verification/kd`

Request:
- `rows[]`:
  - `rowId`
  - `values: { "<НазваниеКолонки>": "<Значение>" }`
- `typeRules[]`:
  - `type`
  - `condition`

Response:
- `dxf`:
  - `blockName`, `isSuccess`, `message`, `issues[]`
- `pdf`:
  - `blockName`, `isSuccess`, `message`, `issues[]`
- `issues[]`:
  - `rowId`
  - `detailName`
  - `severity`: строковый enum (`missing` | `duplicate`)
  - `paths[]`

## 4. Требования к UI
1. В отчёте выводить отдельно списки отсутствующих DXF/PDF.
2. Дубликаты выводить отдельным списком с путями.
3. Подсветка ячеек в таблице:
   - `missing` — класс отсутствия;
   - `duplicate` — класс дубликата.
4. При неизвестном формате `severity` UI не должен «ломать» отчёт.

## 5. Нефункциональные требования
1. Проверка должна отрабатывать для больших таблиц без зависаний UI.
2. Сообщения и статусы должны быть детерминированными и повторяемыми.
3. Логика сериализации enum в API должна быть явно зафиксирована (строки, не числа).

## 6. Критерии приёмки
1. При реальном отсутствии DXF/PDF отчёт показывает имена в нужной колонке (не `—`).
2. При дубликатах появляются записи в блоке дубликатов и подсветка строк.
3. `severity` в JSON приходит строкой, и UI корректно строит `missingByBlock`/`duplicates`.
