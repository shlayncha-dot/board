# ТЗ: доработка меню «Заказ на производство»

## 1. Цель
Упростить ввод данных по наряду и номенклатурам, а также стандартизировать работу со спецификациями для каждой номенклатуры в заказе на производство.

## 2. Изменения в интерфейсе

### 2.1. Единый блок для имени наряда и добавления номенклатуры
Необходимо переработать форму так, чтобы:
- поле **«Имя наряда»**;
- кнопка **«Добавить номенклатуру»**;

находились в **одном визуальном поле/блоке** (в одной строке интерфейса).

#### Ожидаемое поведение
- Пользователь вводит/редактирует имя наряда и сразу рядом может добавить номенклатуру.
- Элементы выровнены и читаются как единый функциональный блок создания/редактирования заказа.

## 3. Правила по спецификациям для номенклатуры

### 3.1. Автосоздание 4 значений спецификации
Для **каждой** добавленной номенклатуры система должна создавать 4 значения спецификаций (типы):
1. `Basic`
2. `Wire`
3. `Packaging`
4. `Tech`

### 3.2. Источник данных
Спецификации должны браться **из БД**, в которую ранее загружаются спецификации по типам.

Требование:
- для каждого из типов (`Basic`, `Wire`, `Packaging`, `Tech`) система подтягивает доступные спецификации из БД;
- в таблице должно отображаться **имя спецификации** (а не только ID/код).

### 3.3. Выбор текущей спецификации после загрузки
После загрузки значений спецификаций в таблице пользователь должен иметь возможность указать **текущую спецификацию** для каждого типа.

#### Ожидаемое поведение
- В строке номенклатуры присутствуют 4 поля/ячейки (или 4 подстроки) для типов `Basic/Wire/Packaging/Tech`.
- В каждом типе доступен выбор из загруженных из БД спецификаций соответствующего типа.
- Выбранное значение сохраняется как текущее для конкретной номенклатуры.

## 4. Функциональные требования
1. При добавлении новой номенклатуры автоматически инициализируются 4 типа спецификаций: `Basic`, `Wire`, `Packaging`, `Tech`.
2. Для каждого типа выполняется запрос к БД за списком спецификаций по соответствующему типу.
3. В UI-таблице показывается наименование спецификации.
4. Пользователь может изменить/выбрать текущую спецификацию в каждой из 4 категорий.
5. Выбранные текущие спецификации сохраняются и корректно восстанавливаются при повторном открытии заказа.

## 5. Нефункциональные требования
- Время загрузки списка спецификаций должно быть приемлемым для работы пользователя (без заметных зависаний интерфейса).
- При отсутствии спецификаций по конкретному типу в БД интерфейс должен показывать понятное состояние (например: «Нет доступных спецификаций»).

## 6. Проверка и критерии приемки
1. Поле «Имя наряда» и кнопка «Добавить номенклатуру» отображаются в одном блоке/строке.
2. После добавления номенклатуры в интерфейсе присутствуют 4 типа спецификаций: `Basic`, `Wire`, `Packaging`, `Tech`.
3. Для каждого типа выводятся спецификации из БД, загруженные ранее по этому типу.
4. В таблице отображается имя выбранной спецификации.
5. Пользователь может выбрать текущую спецификацию по каждому из 4 типов и сохранить выбор.

## 7. Открытые вопросы (на уточнение)
1. Что считать «одним полем»: строго одна строка формы или единый контейнер/панель в одном визуальном блоке?
2. Нужны ли значения по умолчанию (автовыбор) для `Basic/Wire/Packaging/Tech`, если в БД найдено несколько спецификаций?
3. Должна ли система запрещать сохранение номенклатуры, если хотя бы по одному из 4 типов не выбрана текущая спецификация?
4. Нужна ли история смены текущей спецификации и аудит изменений?
