# Windows indexer: установка и запуск

## Что входит
- `tools/windows-indexer/indexer.ps1` — основной скрипт индексации.
- `tools/windows-indexer/run-indexer.bat` — запуск скрипта.
- `tools/windows-indexer/install-autostart-task.bat` — добавление автозапуска через Планировщик задач.
- `tools/windows-indexer/config.json.example` — пример конфигурации.

## Настройка
1. Скопируйте папку `tools/windows-indexer` на удаленный Windows-ПК.
2. В этой папке создайте `config.json` на основе `config.json.example`.
3. Заполните параметры:
   - `serverUrl`: адрес сервера (`https://sls-planning.omnic.pro`).
   - `syncEndpoint`: endpoint API (`/api/file-index/sync`).
   - `scanRoot`: папка для сканирования относительно расположения батника, например `.`.
   - `scanIntervalSeconds`: интервал сканирования.
   - `machineId`: уникальный идентификатор ПК.
   - `maxPayloadBytes` (опционально): максимальный размер одного JSON-запроса к `/api/file-index/sync`. По умолчанию `800000` (~0.8 MB).
   - `requestTimeoutSeconds` (опционально): таймаут HTTP-запроса. По умолчанию `30`.
   - `auth` (опционально): нужен **только** если ваш прокси/сервер требует HTTP-авторизацию.

## Запуск вручную
1. Откройте `cmd` от имени пользователя, под которым должен работать процесс.
2. Выполните:
   - `run-indexer.bat`

Если запускаете из **PowerShell**, используйте:
- `./run-indexer.bat` (или `\.\run-indexer.bat` в Windows-нотации)

> В PowerShell команда из текущей папки без префикса `./` не выполняется по умолчанию.

Пример корректного `config.json`:

```json
{
  "serverUrl": "https://sls-planning.omnic.pro",
  "syncEndpoint": "/api/file-index/sync",
  "scanRoot": ".",
  "scanIntervalSeconds": 30,
  "machineId": "PC-REMOTE-01",
  "maxPayloadBytes": 800000,
  "requestTimeoutSeconds": 30
}
```

`serverUrl` должен быть **строкой**, а не вложенным объектом.


### Что вводить в `auth`
По умолчанию — **ничего**. Если авторизация не нужна, блок `auth` просто не добавляйте в `config.json`.

Добавляйте `auth` только в одном из вариантов:

**Вариант 1: API-ключ в заголовке**

`apiKeyHeader` — это **имя заголовка, которое требует ваш сервер/прокси**.
Частые варианты: `X-API-Key` или `Authorization` (тогда в `apiKey` обычно передают строку вида `Bearer <token>`).
```json
{
  "auth": {
    "apiKeyHeader": "X-API-Key",
    "apiKey": "<ваш_ключ>"
  }
}
```

**Вариант 2: Basic Auth**
```json
{
  "auth": {
    "username": "<логин>",
    "password": "<пароль>"
  }
}
```

Не смешивайте оба варианта одновременно.

Если не знаете, какой именно заголовок использовать — уточните у того, кто настраивал сервер/NGINX/API gateway для `/api/file-index/sync`.


## Как работает синхронизация
- Индексатор сканирует **все файлы** в `scanRoot` рекурсивно (не только PDF/DXF).
- Для каждого файла отправляются: `FileName`, `RelativePath`, `Extension`, `LastWriteTimeUtc`, `SizeBytes`.
- Вычисляется `SnapshotHash` по полному списку файлов.
- Если хэш не изменился относительно предыдущего успешного цикла — отправка не выполняется.
- Если хэш изменился — отправляется новый снимок (при необходимости чанками).

## Частая ошибка: `(413) Request Entity Too Large`
- Это не проблема формата JSON: код `413` означает, что сервер/прокси отклонил слишком большой `POST`-запрос.
- Индексатор теперь автоматически делит большой снимок на части (`ChunkIndex/TotalChunks`) и отправляет его несколькими запросами.
- Проверьте:
  - что `scanRoot` не указывает на слишком большую директорию (лучше сузить до рабочей папки);
  - что в `config.json` задан `maxPayloadBytes` (например, `400000` или `800000`);
  - что на сервере и reverse-proxy увеличен лимит тела запроса для `/api/file-index/sync` (рекомендуется оставить запас, даже при chunk-загрузке).


## Ошибка парсинга PowerShell (строки вида `ParserError`, `MissingEndParenthesisInMethodCall`)
Если в консоли появляются «красные» ошибки парсинга на строках с `Contains(...)`, обычно запущена устаревшая/поврежденная копия `indexer.ps1` (часто после перекодировки файла на Windows).

Что сделать:
- перезаписать папку `tools/windows-indexer` свежей версией из репозитория;
- убедиться, что запускается именно `run-indexer.bat` из этой папки;
- не редактировать `indexer.ps1` в редакторах, которые меняют кодировку автоматически.

## Автозапуск (всегда работать)
1. Откройте `cmd` от администратора.
2. Выполните:
   - `install-autostart-task.bat`
3. Скрипт создаст задачу `SLS_FileIndexer`, которая стартует при запуске Windows.

## Компиляция
Для батника и PowerShell-скрипта компиляция **не требуется**. Достаточно скопировать файлы и запустить `.bat`.

## Если окно "висит" сразу после запуска
Это обычно означает одно из двух:
- идёт первый глубокий проход по большой папке (особенно при `scanRoot` = `C:\...`);
- зависает сетевой запрос к серверу/прокси.

В текущей версии индексатор пишет этапы в консоль (`Scan started`, `Files discovered`, `Sending ... chunk(s)`) и использует `requestTimeoutSeconds`, чтобы запрос не висел бесконечно.

Рекомендации:
- временно сузить `scanRoot` до конкретной рабочей папки, а не всего диска;
- проверить доступность `https://.../api/file-index/sync` из этого ПК;
- при необходимости прокси-аутентификации заполнить блок `auth` в `config.json`.

## Нужен ли пароль к базе?
Нет, батник **не подключается к БД напрямую**.
Он отправляет JSON на HTTP endpoint `/api/file-index/sync`, а уже сервер сохраняет снимок у себя.
Пароль нужен только если ваш сервер/прокси требует авторизацию на уровне HTTP (см. блок `auth`).

## Тестовий батник (перевірка "1 файл -> Дякую")
Доданий готовий файл `test-indexer.bat` у корені репозиторію.

Як запускати на Windows:
1. Відкрийте `cmd`.
2. Перейдіть в папку, куди ви скопіювали репозиторій:
   - `cd /d D:\SLS-Planning-1C`
3. Запустіть:
   - `test-indexer.bat`

Важливо:
- якщо запускати команду не з папки репозиторію, Windows покаже: `"test-indexer.bat" не является внутренней или внешней командой`;
- у такому випадку запускайте повним шляхом, наприклад:
  - `D:\SLS-Planning-1C\test-indexer.bat`
