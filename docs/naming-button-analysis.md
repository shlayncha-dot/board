# Анализ кнопки «Нейминг» и команд в 1С (через API)

## Что делает кнопка «Нейминг»
1. В UI (KD Check) кнопка вызывает `onRunNamingCheck`.
2. Клиент формирует payload из Excel-строк и отправляет POST в backend на `/api/verification/naming`.
3. Backend проксирует проверку во внешний API 1С (`ExternalApis:Naming:CheckUrl`).

## Какие команды (HTTP-запросы) реально уходят

### 1) Клиент -> backend
- Метод: `POST`
- URL: `/api/verification/naming`
- Content-Type: `application/json`
- Тело:

```json
{
  "items": [
    { "rowId": "12", "name": "Болт М8х20" },
    { "rowId": "13", "name": "Шайба 8" }
  ]
}
```

### 2) Backend -> 1С API
- Метод: `POST`
- URL: `https://erp-ge.omnic.solutions/stainless-dev/hs/sls/nomenclature/check` (из `appsettings.json`)
- Content-Type: `application/json`
- Авторизация: `Basic <base64(username:password)>`
- Тело (важно: без `rowId`, только имя):

```json
[
  { "name": "Болт М8х20" },
  { "name": "Шайба 8" }
]
```

## Почему кнопка могла «не работать»

Основная причина обнаружена в фильтрации типов на клиенте:
- проверяются только строки, где колонка `Тип` строго равна одному из значений: `компл`, `крепеж`, `крепеж_св`.
- если в Excel встречались варианты вроде `Компл.`, `Крепеж св`, `Крепеж-св`, лишние пробелы и т.п., строки отбрасывались, и запрос в backend мог уходить пустым.

## Что изменено
- Улучшена нормализация значения поля `Тип` перед фильтрацией:
  - нижний регистр;
  - замена `ё -> е`;
  - схлопывание пробелов в `_`;
  - удаление пунктуации;
  - унификация дефисов в `_`.

Это делает кнопку устойчивее к типичным Excel-вариантам ввода и снижает случаи, когда «Нейминг» визуально нажимается, но фактически не проверяет нужные строки.
