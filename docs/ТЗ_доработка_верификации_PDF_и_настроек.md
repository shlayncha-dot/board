# Техническое задание: доработка верификации PDF и сохранения параметров

## 1. Цель
Устранить расхождения в логике отчёта верификации КД и сделать параметры верификации постоянными (сохранение на сервере), чтобы:
- в отчёте по PDF выводились **обозначения** деталей, а не значения из колонки «Наименование»;
- параметры из меню **Настройки → Параметры верификации** сохранялись и восстанавливались между сессиями;
- выбор ТИП для PDF выполнялся по значениям из столбца **«Условие»** (перечень значений через запятую).

---

## 2. Область изменений

### 2.1 Back-end
- Модуль верификации КД (PDF-часть).
- API и хранилище пользовательских настроек верификации.

### 2.2 Front-end
- Экран «Настройки» в разделе КД.
- Инициализация параметров верификации при открытии экрана.
- Передача актуальных параметров в запрос верификации.

---

## 3. Функциональные требования

### FR-1. Отчёт PDF должен показывать «Обозначение»
1. При формировании `issues` для блока `PDF` поле `detailName` должно заполняться значением из колонки **«Обозначение»**.
2. Значение из колонки «Наименование» для PDF-ошибок не использовать.
3. Поведение одинаково для статусов:
   - `missing` (не найден чертёж);
   - `duplicate` (найдено более одного файла).

**Ожидаемый результат:** в модальном отчёте и подсветке по PDF отображаются именно обозначения тех деталей, для которых не найден/дублируется чертёж.

---

### FR-2. Сохранение параметров верификации на сервере
1. Параметры из блока **«Настройки → Параметры верификации»** должны сохраняться в серверном хранилище.
2. При повторном открытии страницы/приложения параметры должны подгружаться с сервера.
3. Сохраняемая структура для каждой строки ТИП:
   - `type` (1..5);
   - `description` (строка);
   - `condition` (строка).
4. Кнопка «Сохранить» должна вызывать серверный API сохранения.
5. Кнопка «Отмена» должна откатывать изменения к последнему сохранённому серверному состоянию (а не только локальному состоянию до текущего редактирования).

---

### FR-3. Определение ТИП по столбцу «Условие»
1. Поиск подходящего ТИП должен выполняться по полю `condition` соответствующей записи параметров.
2. В `condition` пользователь указывает список значений **через запятую**.
3. Алгоритм сопоставления:
   - разбить `condition` по запятой `,`;
   - обрезать пробелы у каждого элемента;
   - игнорировать пустые элементы;
   - сравнивать без учёта регистра.
4. Если значение найдено в `condition` конкретного ТИП — использовать алгоритм этого ТИП.
5. Если ни одно условие не подошло — использовать дефолтный ТИП (уточнить в реализации: ТИП 1).

**Примечание:** приоритет совпадений — сверху вниз по списку ТИП (первое совпадение побеждает).

---

## 4. API-контракты

### 4.1 Получение параметров верификации
`GET /api/verification/settings`

**Response 200**
```json
{
  "typeRules": [
    { "type": 1, "description": "...", "condition": "A,B,C" },
    { "type": 2, "description": "...", "condition": "..." }
  ]
}
```

### 4.2 Сохранение параметров верификации
`PUT /api/verification/settings`

**Request**
```json
{
  "typeRules": [
    { "type": 1, "description": "...", "condition": "A,B,C" },
    { "type": 2, "description": "...", "condition": "..." }
  ]
}
```

**Response 200**
- Возврат сохранённых данных в том же формате.

### 4.3 Запуск верификации КД
`POST /api/verification/kd`

- Оставить текущий контракт запуска верификации.
- Для PDF-issues вернуть `detailName = обозначение`.

---

## 5. Требования к данным и хранению
1. Сервер хранит настройки как минимум на уровне приложения (persist между рестартами приветствуется; формат — JSON-файл или БД по текущей архитектуре проекта).
2. При отсутствии сохранённых настроек сервер возвращает дефолтные 5 строк ТИП (1..5) с пустыми `description` и `condition`.
3. Валидация входных данных:
   - `type` уникален;
   - допустимые значения `type`: 1..5;
   - `description` и `condition` могут быть пустыми строками.

---

## 6. Изменения UI/UX
1. При открытии вкладки «Настройки» выполнить загрузку параметров с сервера.
2. Показать индикатор загрузки/ошибки при недоступности API.
3. После успешного сохранения показать подтверждение (toast/alert) и обновить локальное «saved state» из ответа сервера.
4. В отчёте проверки PDF список «Не найдено» и «Дубликаты» должен отображать обозначения.

---

## 7. Нефункциональные требования
1. Обратная совместимость: существующий запуск проверки КД не должен ломаться.
2. Производительность: разбор `condition` не должен существенно влиять на время проверки.
3. Надёжность: при ошибке сохранения настройки не перезаписывать локально как «успешно сохранённые».

---

## 8. Критерии приёмки
1. Для строки с `Обозначение = ABC.123`, если PDF не найден, в отчёте отображается `ABC.123`, а не значение из «Наименование».
2. После изменения параметров верификации и нажатия «Сохранить»:
   - перезагрузка страницы сохраняет внесённые значения;
   - значения приходят с сервера через `GET /api/verification/settings`.
3. Если в `condition` указано `MR, AB, cd`, то для входного префикса `ab` выбирается соответствующий ТИП.
4. Если совпадений по всем `condition` нет — применяется дефолтный ТИП 1.
5. Регрессионно: DXF-проверка работает как раньше.

---

## 9. План реализации (этапы)
1. **Back-end:** добавить DTO/эндпоинты настроек, storage и валидацию.
2. **Back-end:** скорректировать формирование `PDF issues.detailName` на «обозначение».
3. **Front-end:** загрузка/сохранение параметров через API, синхронизация кнопок «Сохранить/Отмена».
4. **Front-end:** проверить отображение отчёта по PDF.
5. **Тестирование:** unit/integration + ручной сценарий на UI.
