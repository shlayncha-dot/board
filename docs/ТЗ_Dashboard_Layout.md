# Техническое задание: Layout для Dashboard (режимы SuperUser и TV Dashboard)

## 1. Цель
Подготовить единый layout для раздела **DashBoard**, который будет использоваться в двух режимах отображения:
1. **SuperUser** — внутри основного меню системы (как рабочий модуль).
2. **Dashboard Screen (TV mode)** — отдельная страница по прямой ссылке для вывода на телевизор/инфопанель.

Основная идея: одна и та же функциональная зона дашборда переиспользуется в двух вариантах оболочки (layout-shell).

---

## 2. Режимы и правила отображения

### 2.1. Режим SuperUser
- Дашборд открывается как пункт основного меню приложения (в существующей навигации).
- Используется стандартный layout приложения для авторизованного режима:
  - верхняя информационная панель;
  - верхнее меню модулей;
  - рабочая область в контентной зоне.
- Контент дашборда вставляется в текущую основную рабочую зону.

### 2.2. Режим Dashboard Screen (TV mode)
- Доступен по отдельной простой ссылке (прямой URL).
- На странице отображается:
  1) только верхняя информационная панель (дата/время, название и т.д. — текущий header-стиль),
  2) сразу под ней — вся основная рабочая зона дашборда.
- Не отображать элементы пользовательского интерфейса, не относящиеся к инфопанели и dashboard-контенту (меню модулей, боковые панели и т.п.).

---

## 3. Архитектурное требование (переиспользование)

1. Реализовать **единый компонент контента дашборда** (условно: `DashboardWorkspace`), содержащий всю рабочую зону.
2. Отображать этот компонент в двух shell-вариантах:
   - в модуле DashBoard внутри основного приложения;
   - на отдельном маршруте TV-mode (минимальный shell: инфопанель + рабочая зона).
3. Запрещено дублировать разметку рабочей зоны в двух разных файлах/экранах.

---

## 4. Layout рабочей зоны Dashboard

Рабочая зона делится **горизонтально на 2 блока**:
1. Верхний блок: **Metalworking**
2. Нижний блок: **Assembly**

Каждый блок делится **вертикально на 2 части**:
- Левая часть: **3/4** ширины блока.
- Правая часть: **1/4** ширины блока.

### 4.1. Левая часть (для обоих блоков)
- Разместить таблицу-заглушку (пока без данных).
- Требования к таблице:
  - есть строка заголовков;
  - фон заголовков — **черный**;
  - текст заголовков — **белый**.

### 4.2. Правая часть (для обоих блоков)
- Разместить секцию с заголовком/подписью: **Statistics**.
- Содержимое пока может быть заглушкой (без бизнес-данных).

---

## 5. Функциональные требования

1. Переключение между режимами не должно менять структуру и содержимое `DashboardWorkspace`.
2. Для TV-mode должна быть отдельная ссылка (route), доступная без перехода через меню.
3. При изменениях в `DashboardWorkspace` отображение должно синхронно обновляться в обоих режимах.

---

## 6. Нефункциональные требования

1. Layout строится адаптивно на CSS Grid/Flex без внешних UI-библиотек.
2. Соотношения 3/4 и 1/4 должны сохраняться стабильно на широких экранах (приоритет — ТВ/desktop).
3. Стиль должен соответствовать текущей визуальной системе проекта.

---

## 7. Критерии приемки

1. В режиме DashBoard (в меню) отображается новая двухуровневая dashboard-структура.
2. По отдельной ссылке открывается TV-mode: верхняя инфопанель + та же рабочая зона дашборда, без лишней навигации.
3. В блоках `Metalworking` и `Assembly`:
   - левая зона 3/4 содержит таблицу с черной шапкой и белым текстом,
   - правая зона 1/4 содержит подпись `Statistics`.
4. Разметка рабочей зоны реализована единым переиспользуемым компонентом.

---

## 8. Этап реализации (текущий)
На текущем этапе требуется создать только layout и заглушки (без бизнес-логики, данных и интеграций API).
